ATDD pipeline — command sequences

Three entry paths depending on what you already have. Each column reads top to bottom. Colored commands are slash commands from the atdd-pipeline plugin. The dashed orange checkpoints are the two human gates.

Case A · existing GitHub issue

The issue already exists (prose or partial Gherkin). Imported without duplication.
/setup-atdd-pipeline
once per repo — tracker / paths / reviewer mode config
→ .atdd-pipeline.json (repo root)
↓
/from-issue 130
fetch gh issue, parse Goal/Acceptance, interview gaps, tag the issue
→ specs/<slug>/context.md, gh labels "atdd,us/<slug>,level/<x>"
↓
/spec-generate
interview ambiguities, generate Gherkin per business rule
→ specs/<slug>/R-*.feature
↓
/spec-review
branches / coherence / gaps / triangulation
→ specs/<slug>/review.md (VERDICT: OK | REGENERATE)
↓
Checkpoint #1 — human
OK → continue · REGEN <reason> → /spec-generate
↓
/to-issues-atdd
creates sub-issues + provisions atdd/<slug>/integration (base for sub-PRs)
→ specs/<slug>/issues.json, gh sub-issues #150…, integration branch
↓
/red <sub-issue>
generate test at the right level, review-fidelity, auto-correct ×2
→ branch atdd/<slug>/scenario-<N> · RED commit
↓
/green <sub-issue>
YAGNI impl, review-architecture ∥ review-intent · draft PR → integration
→ GREEN commit · draft PR base=integration (never main)
↓
/auto-merge <pr>
mark ready · watch CI · idle bot watch · /apply-pr-feedback (×3 max) · squash-merge into integration
→ scenario PR merged · branch deleted · escalation comment on timeout
↓
repeat /red → /green → /auto-merge per sub-issue (sequential, same base)
↓
final PR integration → main
once every scenario is merged into integration, the trunk PR opens
→ gh PR #<N> · labels atdd, us/<slug>, integration
↓
Checkpoint #2 — human
Review & merge the trunk PR (auto-merge refuses this base)

Case B · no GitHub issue (PRD/spec/architecture in hand)

Starting a feature. PRD, ADRs and architecture are clear. No tracker entry yet.
/setup-atdd-pipeline
once per repo
→ .atdd-pipeline.json
↓
/impact-map
interview: actor / goal / action / user story / business rules R-NN
→ specs/<slug>/context.md (reads PRD.md + ADRs as references)
↓
/spec-generate
interview ambiguities, Gherkin with concrete PRD data
→ specs/<slug>/R-*.feature
↓
/spec-review
branches / coherence / gaps / triangulation
→ review.md
↓
Checkpoint #1 — human
↓
/to-issues-atdd
parent + US + scenarios + integration branch (base for sub-PRs)
→ issues.json, gh #1001 parent, #1002 US, #1010+ scenarios
↓
/red <N>
RED test + review-fidelity
↓
/green <N>
GREEN impl + review-architecture ∥ review-intent · draft PR → integration
↓
/auto-merge <pr>
CI watch · bot idle watch · /apply-pr-feedback · squash into integration
↓
repeat /red → /green → /auto-merge per scenario
↓
improve-codebase-architecture (mattpocock)
between scenarios, never during a cycle
→ ADR updated, deeper refactors if patterns emerge
↓
final PR integration → main
opens automatically once integration carries every scenario
↓
Checkpoint #2 — human
review & merge into main

Case C · greenfield (no PRD, no architecture)

New project or exploration. No PRD, no ADRs, no shared language. Everything to bootstrap.
/setup-atdd-pipeline
tracker=local works if no GitHub repo yet
→ .atdd-pipeline.json
↓
grill-with-docs (mattpocock)
bootstrap CONTEXT.md (shared language) + first ADRs
→ CONTEXT.md, docs/adr/0001-*.md
↓
to-prd (mattpocock) — optional
if you want a formal PRD before capturing user stories
→ PRD.md
↓
/impact-map
capture first user story leaning on the fresh CONTEXT.md
→ specs/<slug>/context.md
↓
/spec-generate
heavy interview (little established context)
→ R-*.feature
↓
/spec-review
↓
Checkpoint #1 — human
↓
/to-issues-atdd
tracker=local → .atdd/issues/*.md if no GitHub yet · otherwise provisions the integration branch too
↓
/red <id>
tracer bullet: first nominal scenario of the first US
↓
/green <id>
impl + draft PR → integration (when tracker=github)
↓
/auto-merge <pr>
CI + bots + /apply-pr-feedback + squash · skipped when tracker=local
↓
repeat scenarios · revisit CONTEXT.md often
↓
final PR integration → main
↓
Checkpoint #2 — human
the only manual stop in the pipeline
INPUT / capture SPEC / gherkin SYNC / issues RED / test GREEN / impl AUTO-MERGE / final PR Human checkpoint
Shortcut, one command:
/atdd-run <us-slug>  chains impact-map → spec → checkpoint #1 → to-issues-atdd (+ integration branch) → for each scenario: red → green → auto-merge into integration, then opens the final integration → main PR and stops on checkpoint #2.
/atdd-run --from-stage auto-merge <us-slug>  resumes watching sub-PRs that are already open.
/atdd-run --no-auto-merge <us-slug>  disables auto-merge; falls back to the legacy per-scenario human checkpoint #2.
/atdd-run --sequential <us-slug>  reviewers in series (Codex or low-token budget).
/atdd-run --dry-run <us-slug>  generates scenarios + shows the issue plan without writing anything.
/auto-merge <pr>  watch + apply-pr-feedback + merge standalone. Refuses when baseRefName = trunk.
