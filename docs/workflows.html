ATDD pipeline — séquences de commandes

Trois cas d'entrée selon ce que tu as déjà sous la main. Chaque colonne lit de haut en bas. Les commandes en couleur sont des slash commands du plugin atdd-pipeline. Les checkpoints en pointillé orange sont les deux portes humaines.

Cas A · via issue GitHub existante

L'issue existe déjà (prose ou Gherkin partiel). On l'importe sans la dupliquer.
/setup-atdd-pipeline
une fois par repo — config tracker / paths / mode reviewer
→ .atdd-pipeline.json (root du repo)
↓
/from-issue 130
fetch gh issue, parse Goal/Acceptance, interview gaps, tag l'issue
→ specs/<slug>/context.md, gh label "atdd,us/<slug>,level/<x>"
↓
/spec-generate
interview ambigus, génère Gherkin par règle métier
→ specs/<slug>/R-*.feature
↓
/spec-review
branches / cohérence / trous / triangulation
→ specs/<slug>/review.md (VERDICT: OK | REGENERATE)
↓
Checkpoint #1 — humain
OK → continue · REGEN <raison> → /spec-generate
↓
/to-issues-atdd
crée sub-issues + provisionne atdd/<slug>/integration (base des sub-PRs)
→ specs/<slug>/issues.json, gh sub-issues #150…, branche integration
↓
/red <sub-issue>
gen test au bon niveau, review-fidelity, auto-corr ×2
→ branche atdd/<slug>/scenario-<N> · commit RED
↓
/green <sub-issue>
impl YAGNI, review-archi ∥ review-intent · draft PR → integration
→ commit GREEN · draft PR base=integration (jamais main)
↓
/auto-merge <pr>
ready PR · watch CI · idle bot watch · /apply-pr-feedback (×3 max) · squash-merge dans integration
→ PR scenario merged · branche supprimée · escalation comment si timeout
↓
repeat /red → /green → /auto-merge pour chaque sub-issue (sequentiel, même base)
↓
PR finale integration → main
une fois tous les scenarios merged dans integration, ouverture de la PR vers trunk
→ gh PR #<N> · labels atdd, us/<slug>, integration
↓
Checkpoint #2 — humain
Review & merge la PR vers main (auto-merge refuse cette base)

↓

↓

Cas C · greenfield (pas de PRD ni archi)

Projet neuf ou exploration. Pas de PRD, pas d'ADR, pas de shared language. Tout à construire.
/setup-atdd-pipeline
tracker=local possible si pas de repo GitHub encore
→ .atdd-pipeline.json
↓
grill-with-docs (mattpocock)
bootstrap CONTEXT.md (shared language) + premiers ADRs
→ CONTEXT.md, docs/adr/0001-*.md
↓
to-prd (mattpocock) — optionnel
si tu veux un PRD formel avant de capturer les US
→ PRD.md
↓
/impact-map
capture première US en s'appuyant sur CONTEXT.md frais
→ specs/<slug>/context.md
↓
/spec-generate
interview lourd (peu de contexte établi)
→ R-*.feature
↓
/spec-review
↓
Checkpoint #1 — humain
↓
/to-issues-atdd
tracker:local → .atdd/issues/*.md si pas encore de GitHub · sinon provisionne aussi integration branch
↓
/red <id>
tracer-bullet : 1ère scenario nominal de la 1ère US
↓
/green <id>
impl + draft PR → integration (si tracker=github)
↓
/auto-merge <pr>
CI + bots + /apply-pr-feedback + squash · skip si tracker=local
↓
repeat scenarios · revisit CONTEXT.md souvent
↓
PR finale integration → main
↓
Checkpoint #2 — humain
seul stop manuel du pipeline
INPUT / capture SPEC / gherkin SYNC / issues RED / test GREEN / impl AUTO-MERGE / PR finale Checkpoint humain
Raccourci — tout en un :
/atdd-run <us-slug>  chaîne impact-map → spec → checkpoint #1 → to-issues-atdd (+ integration branch) → pour chaque scenario : red → green → auto-merge dans integration, puis ouvre la PR finale integration → main et stoppe sur le checkpoint #2.
/atdd-run --from-stage auto-merge <us-slug>  reprend au watch des sub-PRs déjà ouvertes.
/atdd-run --no-auto-merge <us-slug>  désactive l'auto-merge ; rebascule sur l'ancien checkpoint #2 humain par scenario.
/atdd-run --sequential <us-slug>  reviewers en série (Codex ou budget tokens).
/atdd-run --dry-run <us-slug>  génère scenarios + montre plan issues sans rien écrire.
/auto-merge <pr>  watch + apply-pr-feedback + merge en standalone. Refuse si baseRefName = trunk.
